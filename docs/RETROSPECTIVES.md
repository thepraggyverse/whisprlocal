# Milestone Retrospectives

Append-only log of what went well, what surprised, and one specific lesson per milestone.

## M0 — Project skeleton (shipped 2026-04-17, v0.1.0-m0 @ 2add398)

What went well: the plan-before-code-before-commit cadence kept every failure local. The Step 5a MLX/WhisperKit discovery pass caught that `mlx-swift-examples` no longer exports `MLXLLM` — exactly the kind of assumption that would have produced a load-bearing bug at M3 if pinned blindly at M0. Gate 4.5's link-surface check proved the keyboard architecture holds by construction, not by convention.

Surprises: entitlements files got silently clobbered when xcodegen's `entitlements:` key lacked inline `properties:` — a five-minute recovery but unexpected. The iPhone 16 Pro → iPhone 17 Pro drift was a sharper reminder that "latest" means different things to different Xcode installs. CI caught the `rg`-not-on-macos-15 gap I should have predicted at the workflow-authoring moment.

For M1: verify tool availability on the target environment before writing the code that depends on it — check `gh api repos/actions/runner-images/contents/images/macos` (or equivalent) the first time I reach for any CLI tool in CI, not after the red X.

Post-CI-green bug caught via simulator verification: `CaptureError` code 2 on record tap. Root cause: `CODE_SIGNING_ALLOWED: NO` in Debug stripped the App Group entitlement from the binary. CI didn't catch it because tests stubbed the inbox URL provider. Fix: re-enable signing on Debug (simulator uses free ad-hoc identity), add regression test that asserts `AppGroupPaths.containerURL` is non-nil, add CI step that greps the binary for the entitlement, codify in CLAUDE.md anti-patterns.

Lesson: "green CI" is necessary but not sufficient. The simulator verification gate exists precisely for entitlement/runtime-config bugs that test suites (by their nature) mock away. This case validates the "PR stays draft until human verifies on simulator" rule from Mode 2.

### Lessons codified back into project rules
- [x] Add to CLAUDE.md anti-patterns: "Do not use xcodegen `entitlements:` key without inline `properties:` — silently clobbers hand-written entitlements files. Use `CODE_SIGN_ENTITLEMENTS` build setting to reference them instead." — applied at M1 kickoff.
- [x] Add to CLAUDE.md session workflow: "Before writing CI code that depends on a CLI tool, verify the tool is preinstalled on the target runner image (check the macos-15 image manifest at github.com/actions/runner-images)." — applied at M1 kickoff.
- [x] Add to CLAUDE.md build commands: "Sim destination uses `OS=latest,name=iPhone <N> Pro` where N is the highest Pro tier shipped with the installed Xcode." — was already present in CLAUDE.md during M0 (lines 63-69); no new edit needed.

## M1 — Audio capture (shipped 2026-04-17, branch `milestone/M1-audio`)

What went well: the commit-sequence plan held up end-to-end — every commit compiled and tested in isolation, and each `/verify` loop caught one real issue (the file-protection simulator-vs-device gap) exactly where it was cheapest to fix. Splitting WAVWriter and AudioConverter into two small files made the engine-tap closure in AudioCaptureService boring — it literally just wires references together, which is what code on the audio I/O thread should look like. Delegating RIFF/WAVE header correctness to `AVAudioFile` dodged a whole class of IEEE-float-vs-PCM header bugs that would have been silent on the test WAVs and loud on the first real WhisperKit invocation at M2.

Surprises: iOS simulator does not honor `NSFileProtectionComplete` — `attributesOfItem` returns `nil` for `.protectionKey` because the sim is running on macOS, which doesn't implement iOS Data Protection. The fix was simple (tolerate `nil` on simulator, assert `.complete` on device), but the failure mode was "your privacy invariant is invisible in tests" — exactly the kind of thing to surface in PRIVACY.md. The second surprise: the M0 `AudioFormat.swift` docstring had already been partly wrong (it claimed the keyboard did the conversion) — caught it during commit 2 before building on the misconception. Always read scaffolding the previous milestone left you before reusing it.

For M2: Apple's simulator gaps around privacy attributes are a systematic testability hole, not a one-off. When M2 adds HF model downloads, assume the sim will not enforce ATS exactly the way the device does either — plan a manual device verification step into the milestone, not just CI.

### Lessons codified back into project rules
- [ ] Add to docs/PRIVACY.md (near file-protection paragraph): "simulator tolerance is expected; device-strict assertions live in the test suite." — applied in this commit.
- [ ] Add to CLAUDE.md anti-patterns: "Do not assume iOS Data Protection, ATS, or keychain behavior on the simulator matches device. Privacy-invariant tests should be simulator-tolerant with a device-strict branch." — propose as a follow-up for M2 planning rather than rushing here.

## M2 — WhisperKit integration (shipped 2026-04-17, branch `milestone/M2-whisperkit`)

What went well: the Step 5a discovery pass paid off twice. First by surfacing the upstream repo rename (`WhisperKit` → `argmax-oss-swift`) and the three open known-issues (#392 suppressTokens, #315 Swift 6 prewarm, #408 Xcode 26 scan warnings) before any code landed — producing ADR-002 and turning mitigations into grep-able code comments rather than tribal knowledge. Second by forcing an explicit version pin (`.exact(0.18.0)` instead of `.upToNextMinor`) that will make future bumps deliberate review events. The 8-commit architecture held: data layer → engine → service → UI → integration-test → docs, each independently compilable, and `/verify` caught at least one real issue per commit (lint limits after extracting diagnostics, env-var-vs-compile-time gate for the integration test, Package.resolved trackability). Splitting `InboxJobWatcher` from `AudioCaptureService` via the App Group inbox/envelope/Darwin notification contract (even though the keyboard doesn't exist yet) means M4 is genuinely a drop-in rather than a refactor.

Surprises, in ascending order of "glad we caught it before ship":

- **Shell-env vs test-runner env.** The opt-in integration-test gate started as `ProcessInfo.processInfo.environment["WHISPR_INTEGRATION"] == "1"` with an `XCTSkipUnless`. That pattern silently skips even when the user sets the env var, because `xcodebuild test` does not propagate shell env to the iOS Simulator test-runner process. Switched to `#if WHISPR_INTEGRATION` gated on `OTHER_SWIFT_FLAGS='$(inherited) -D WHISPR_INTEGRATION'` — only reliable CLI-driven gate for this invocation shape. Lesson: when a gate "should" work, prove it by watching the log lines, not by assuming the XCTSkip message means "ran-and-skipped-correctly."
- **Xcode extracts diagnostics into "iPhone 18.0 is not installed".** Midway through the milestone, CI flipped from green to red across a window of ~36 minutes on the same commit, same runner class. Root cause: GitHub rotated the `macos-15` image and the new image shipped without the iOS Simulator runtime. Our CI's `xcodebuild build ... | tail -60` was masking the nonzero exit because the pipe swallowed the destination-validation error, so the "Build main app" step reported success and the "Verify App Group entitlement is embedded" step was the one that finally blew up (on a missing `.app` bundle). Noting this for M4+ CI hardening: pin image digests or install the runtime explicitly, and drop the `| tail -60` so step exits propagate.
- **Download path vs load path.** WhisperKit v0.18's `download(variant:downloadBase:)` returns a deep URL `<downloadBase>/models/<repo>/<variant>/`, but `WhisperKitConfig.modelFolder` expects that same deep URL as its load-time input. We were passing the root. Every model variant failed identically at load with `modelsUnavailable("Model file not found at …/MelSpectrogram.mlmodelc")`. The fix was surgical — thread the deep URL via a resolver closure — and the regression guard (`testEngineRejectsUnDownloadedModel`) is cheaper than the sim check that caught it. Lesson: when an upstream API exposes two path-shaped fields with similar names, read the source for one usage each, not just the most obvious one.
- **Unit tests say yes, live mic says empty.** The pipeline passed 75 unit tests, a 25-second real-WhisperKit round-trip against a silent procedurally-generated WAV, and a JFK WAV injection test that rendered the correct transcript in the UI. Live-mic recording on an iPhone 17 Pro Max (iOS 26.4, WhisperKit 0.18) still returns empty transcripts in ~20 ms. Voice Memos works on the same device, so the hardware is fine. Simulator live-mic is broken by an iOS 26 HAL audio bug independent of our code. Diagnostic instrumentation sits on `fix/m2-audio-diagnostics` (commit 2960c2d) ready to run when we pick this back up.

### Verified vs deferred

**Verified at M2:**
- Unit tests: 75 passing on `milestone/M2-whisperkit`, covering model catalog invariants, ADR-002 mitigation guards, download/resolve contract, inbox watcher pair-matching + cleanup, envelope handoff + Darwin-notification spy.
- Integration test (`WhisperEngineIntegrationTests`, opt-in via `-D WHISPR_INTEGRATION`): real model download, real Core ML load, real transcribe — 25 seconds end-to-end, zero failures. Deep-URL contract asserted (`MelSpectrogram.mlmodelc` exists in the resolved folder).
- Simulator WAV injection: dropping a known-good WAV into App Group `inbox/` → `InboxJobWatcher` → correct transcript rendered in the Record tab. Transcription pipeline is correct.

**Deferred to M4 (or M7, whichever triggers first):**
- Live-mic verification on simulator and real device. Simulator live-mic is broken by an iOS 26 HAL audio bug (Apple's, not ours). Real device (iPhone 17 Pro Max, iOS 26.4, WhisperKit 0.18) returns empty transcripts in ~20 ms for `whisper-tiny.en` and `whisper-small`. Voice Memos works on the same device → mic hardware is fine; the empty-transcript symptom is in the capture-to-engine handoff, not in inference. Three code-level validations (unit tests, integration test, WAV injection test) prove the transcription pipeline is correct. The capture side needs deeper investigation — `fix/m2-audio-diagnostics` (commit 2960c2d) has full instrumentation staged: per-tap RMS + peak + first-8-samples, post-WAV stats + re-read RMS, AVAudioSession state, DEBUG-gated inbox retention for offline inspection. Deferring live-mic debugging to M4 (keyboard extension forces device testing anyway) or M7 (launch prep). Simulator dev flow with WAV injection is adequate for M3–M6 milestones.

For M3: MLX-LLM polish lands on top of the verified pipeline; every transcript the polish engine sees has already round-tripped through `WhisperEngine` end-to-end, so M3 can focus on prompt templates and raw/polished UI without re-validating the base pipeline. Two concrete carryovers: (a) the opt-in compile-time-gated integration pattern (`#if WHISPR_INTEGRATION`) is the template for M3's real-MLX round-trip; (b) ADR-002's "revisit when" list should get re-read before picking the MLX path so we capture any changes in that ecosystem the way we did for WhisperKit.

### Lessons codified back into project rules
- [ ] Add to CLAUDE.md build commands or anti-patterns: "Gate opt-in tests with `#if <FLAG>` + `OTHER_SWIFT_FLAGS='$(inherited) -D <FLAG>'` rather than `XCTSkipUnless(ProcessInfo.environment…)`. xcodebuild does not propagate shell env to the iOS Simulator test-runner; env-var gates silently skip." — propose in M3 planning.
- [ ] Add to CI workflow (separate PR `ci/sim-runtime-hardening`): install the iOS Simulator runtime explicitly rather than relying on whatever ships in `macos-15`; drop `| tail -60` after `xcodebuild build` so step exits propagate; consider pinning the runner image digest. — propose as a follow-up before M3 kickoff.
- [ ] Add to docs/PRIVACY.md (M2 addendum): model weights live in `Application Support/Models/` (out of iCloud backup, out of user-visible Files). Audio stays in App Group `inbox/` transiently and is deleted after transcription on both success and failure (§8.6). — already in ARCHITECTURE.md M2 section; reiterating in PRIVACY.md is a nice-to-have for M7 hardening, not urgent.
- [ ] Add to CLAUDE.md "When you're unsure": for any API surface that exposes two similarly-named path fields (`downloadBase` vs `modelFolder`, etc.), read the source for each separately before using either — conflation produces runtime failures that unit tests with mocks cannot catch. — propose in M3 planning.
